# /transform-roadmap -- 아웃컴 중심 로드맵

계획된 기능 목록 또는 아웃풋 중심 로드맵을 가져다가 *무엇*을 하는지 대신 *왜* 하는지를 전달하는 아웃컴 중심 로드맵으로 재작성합니다.

## 사용 예시

```
/transform-roadmap [기능 목록 또는 로드맵 붙여넣기]
/transform-roadmap [로드맵 문서, 스프레드시트, 또는 스크린샷 업로드]
```

## 워크플로우

### 1단계: 현재 로드맵 수락

다음 형식으로 수락합니다:
- 기능 목록 또는 백로그 아이템
- 로드맵 문서 (Now/Next/Later, 분기별, 타임라인)
- 스프레드시트 또는 간트 차트 내보내기
- 로드맵 도구의 스크린샷

각 항목을 파싱하여 기능 이름, 설명, 목표 날짜/기간, 맥락을 추출합니다.

### 2단계: 전략적 맥락 파악

물어봅니다:
- 이 기간의 제품 목표 또는 OKR은 무엇인가요?
- 이 로드맵의 대상은 누구인가요? (임원, 엔지니어링, 고객, 이사회)
- 어떤 형식을 선호하시나요? (Now/Next/Later, 분기별, 타임라인)

### 3단계: 각 항목 변환

**outcome-roadmap** 스킬 적용:

로드맵의 각 기능/아웃풋에 대해:
1. 달성하려는 **사용자 또는 비즈니스 아웃컴** 파악
2. 아웃컴 문장으로 재작성: "[동사] [메트릭/경험] for [사용자 세그먼트]"
3. 동일한 아웃컴을 달성하는 기능들을 하나의 이니셔티브로 그룹화
4. 각 아웃컴에 성공 메트릭 추가

**변환 전 → 후 예시:**
- "SSO 통합 개발" → "기업 온보딩 마찰 감소 — 목표: 기업 고객 첫 가치 달성 시간 50% 단축"
- "대시보드 재설계" → "파워 유저가 더 빠르게 인사이트를 찾을 수 있도록 지원 — 목표: 인사이트 도달 시간 30% 감소"
- "CSV 내보내기 추가" → "팀이 제품 외부에서 데이터를 공유할 수 있도록 지원 — 목표: 보고서 공유 25% 증가"

### 4단계: 변환된 로드맵 생성

```
## 아웃컴 중심 로드맵: [제품] — [기간]

**전략적 테마**: [2-3개의 상위 테마]

### Now (현재 분기)
**테마: [전략적 테마]**
| 아웃컴 | 성공 메트릭 | 주요 이니셔티브 | 상태 |
|---------|---------------|----------------|--------|

### Next (다음 분기)
**테마: [전략적 테마]**
| 아웃컴 | 성공 메트릭 | 주요 이니셔티브 | 신뢰도 |
|---------|---------------|----------------|------------|

### Later (미래)
**테마: [전략적 테마]**
| 아웃컴 | 성공 메트릭 | 주요 이니셔티브 | 의존성 |
|---------|---------------|----------------|-------------|

### 변환 노트
| 원래 기능 | 변환된 아웃컴 | 이 구성을 선택한 이유 |
|-----------------|--------------------|-----------------|

### 변경된 것들
[로드맵 내러티브가 어떻게 변화했는지 요약]
```

마크다운 파일로 저장합니다.

### 5단계: 검토

제안합니다:
- "각 아웃컴에 **OKR 정렬 정보를 추가**해 드릴까요?"
- "이 로드맵의 **이해관계자 프레젠테이션을 작성**해 드릴까요?"
- "Now 항목들에 대한 **리스크를 파악**해 드릴까요?"

## 참고사항

- 아웃컴은 측정 가능하고 명확한 "완료" 상태가 있어야 합니다
- 여러 기능이 하나의 아웃컴에 매핑될 수 있습니다 — 이것은 버그가 아닌 기능입니다
- 아웃풋이 아웃컴을 명확히 달성하지 않는 경우, 사용자에게 정당화하거나 우선순위를 낮추도록 표시하세요
- 대상이 중요합니다: 임원 로드맵은 아웃컴만, 엔지니어링 로드맵은 각 아웃컴 하에 결과물을 포함할 수 있습니다
- "Later" 항목은 덜 구체적이어야 합니다 — 확정된 솔루션 없는 아웃컴으로
