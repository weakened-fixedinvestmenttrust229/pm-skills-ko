# /sprint -- 스프린트 라이프사이클

스프린트 라이프사이클을 다루는 세 가지 모드: 스프린트 플래닝을 위한 **plan**, 회고를 위한 **retro**, 출시 커뮤니케이션을 위한 **release-notes**.

## 사용 예시

```
/sprint plan 2주 스프린트, 엔지니어 4명, 체크아웃 개선 집중
/sprint retro [팀 피드백 또는 스프린트 데이터 붙여넣기]
/sprint release-notes [티켓, 변경 로그, 또는 PRD 붙여넣기]
/sprint                    # 현재 어느 단계인지 물어봄
```

## 모드

---

### Plan 모드

용량 추정, 스토리 선택, 리스크 파악으로 스프린트 플래닝을 준비합니다.

#### 워크플로우

**1단계: 스프린트 맥락 수집**
- 스프린트 기간 (1주 또는 2주)
- 팀 구성 (엔지니어, 디자이너, QA — 및 가용성)
- 스프린트 목표 또는 집중 영역
- 고려할 백로그 아이템 (붙여넣기, 업로드, 또는 설명)
- 지난 스프린트에서 이월된 항목
- 알려진 방해 요소 (휴일, 온콜, 회의)

**2단계: 용량 추정**

**sprint-plan** 스킬 적용:

- 회의, 온콜, PTO 후 사용 가능한 엔지니어링 시간/포인트 계산
- 과거 데이터 기반 속도 조정 (제공된 경우) 또는 업계 표준 (이론적 용량의 70%) 적용
- 팀원별 용량 세분화 표시

**3단계: 스토리 선택 및 순서 결정**

- 용량 내에서 적합한 스토리 추천
- 의존성 체인 표시 (A가 완료되어야 B가 시작 가능)
- 리스크 파악: 스펙이 불충분하거나, 외부 의존성이 있거나, 디자인 입력이 필요한 스토리
- 빠른 성과와 대형 아이템의 균형 유지
- 모든 스토리에 인수 기준이 있는지 확인

**4단계: 스프린트 계획 생성**

```
## 스프린트 계획: [스프린트 이름/번호]

**기간**: [날짜]
**스프린트 목표**: [한 문장]
**팀**: [팀원 및 가용성]

### 용량
| 팀원 | 가용 일수 | 포인트/시간 | 비고 |
|--------|--------------|-------------|-------|

**총 용량**: [X] 포인트/시간
**권장 커밋**: [Y] 포인트/시간 (버퍼 포함)

### 선택된 스토리
| # | 스토리 | 포인트 | 담당자 | 의존성 | 리스크 |
|---|-------|--------|-------|-------------|------|

### 스프린트 리스크
1. [리스크] — 완화: [액션]

### 완료 기준
- [ ] 코드 리뷰됨
- [ ] 테스트 통과
- [ ] 스테이징에 배포됨
- [ ] QA 승인됨
- [ ] 문서 업데이트됨 (해당되는 경우)
```

### Retro 모드

실행 가능한 개선 사항을 도출하는 구조화된 회고를 진행합니다.

#### 워크플로우

**1단계: 스프린트 피드백 수집**

다음으로 입력 수락:
- 팀 피드백 (설문, Slack, 또는 협업 문서에서 붙여넣기)
- 스프린트 메트릭 (속도, 버그, 인시던트)
- 사용자 자신의 관찰

물어봅니다: "어떤 회고 형식을 선호하시나요?"
- **Start/Stop/Continue** (단순, 빠름)
- **4Ls** (Liked, Learned, Lacked, Longed for)
- **Sailboat** (Wind = 도움, Anchor = 방해, Rocks = 리스크, Island = 목표)

**2단계: 분석 및 구조화**

**retro** 스킬 적용:

- 피드백을 선택한 프레임워크로 분류
- 주제 및 패턴 파악
- 증상과 근본 원인 분리
- 축하할 만한 성과 강조

**3단계: 회고 요약 생성**

```
## 스프린트 회고: [스프린트 이름]

**날짜**: [오늘]
**형식**: [Start/Stop/Continue | 4Ls | Sailboat]
**참석자**: [알 경우]

### 잘 된 것들
[지지 근거와 함께 그룹화된 주제]

### 잘 안 된 것들
[근본 원인 분석과 함께 그룹화된 주제]

### 핵심 인사이트
[나타난 2-3개의 패턴]

### 액션 아이템
| # | 액션 | 담당자 | 마감일 | 우선순위 |
|---|--------|-------|----------|----------|

### 이번 스프린트 메트릭
| 메트릭 | 이번 스프린트 | 지난 스프린트 | 트렌드 |
|--------|-----------|------------|-------|
```

### Release Notes 모드

기술적 아티팩트에서 사용자 대상 릴리스 노트를 생성합니다.

#### 워크플로우

**1단계: 릴리스 내용 수락**

다음 수락:
- Jira/Linear 티켓 또는 변경 로그
- PRD 또는 기능 스펙
- Git 커밋 메시지 또는 PR 설명
- 팀의 출시된 내용 내부 요약

**2단계: 변환**

**release-notes** 스킬 적용:

- 기술적 언어를 사용자 혜택으로 번역
- 다음으로 분류: 새 기능, 개선 사항, 버그 수정
- 제품의 보이스로 작성 (명확하지 않은 경우 톤 확인)
- 가장 영향력 있는 변경 사항을 먼저 강조

**3단계: 릴리스 노트 생성**

```
## 새로운 기능 — [버전/날짜]

### 하이라이트
[가장 중요한 변경 사항에 대한 1-2문장 요약]

### 새 기능
- **[기능 이름]** — [평이한 언어로 사용자 대상 혜택]

### 개선 사항
- **[개선 사항]** — [무엇이 더 나아졌는지]

### 버그 수정
- [사용자 영향을 야기한 문제] 수정

### 곧 출시 예정
[다음 릴리스 예고 (선택 사항)]
```

마크다운으로 저장하고 다양한 채널을 위한 형식 변환 제안 (블로그 포스트, 인앱, 이메일, Slack 공지).

## 참고사항

- plan 모드: 계획되지 않은 작업을 위해 20% 버퍼를 확보하세요 — 100% 용량으로 계획하는 팀은 항상 놓칩니다
- retro 모드: 아무도 실행하지 않을 10가지가 아닌, 영향력 높은 2-3개의 액션 아이템에 집중하세요
- release notes: 항상 변경 사항을 기술적 구현이 아닌 사용자 혜택으로 구성하세요
- 각 모드는 연결될 수 있습니다: plan → (스프린트 진행) → retro → release-notes
