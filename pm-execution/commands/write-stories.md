# /write-stories -- 백로그 아이템 생성기

기능을 잘 구조화된 백로그 아이템으로 분해합니다. 팀의 선호도에 따라 세 가지 형식 중 선택하며, 각각 완전한 인수 기준을 포함합니다.

## 사용 예시

```
/write-stories user 사용자가 보고서를 PDF 및 CSV로 내보낼 수 있도록 허용
/write-stories job 작업 마감일을 위한 알림 시스템
/write-stories wwa 모바일 앱의 다크 모드
/write-stories [PRD 또는 기능 스펙 업로드]      # 형식 선호도 물어봄
/write-stories                                      # 기능 및 형식 물어봄
```

## 형식

### 사용자 스토리 (User Stories)
**형식**: "As a [사용자 유형], I want [기능], so that [혜택]"
**user-stories** 스킬 적용:
- 3C 원칙 준수: Card (간략함), Conversation (맥락), Confirmation (인수 기준)
- INVEST 기준 검증: Independent, Negotiable, Valuable, Estimable, Small, Testable
- 관련된 경우 디자인 링크 또는 목업 참조 포함

### 잡 스토리 (Job Stories)
**형식**: "When [상황], I want to [동기], so I can [결과]"
**job-stories** 스킬 적용:
- 사용자 역할이 아닌 상황과 맥락에 집중
- 리서치에서 관찰한 실제 사용자 시나리오에 근거
- JTBD 지향 팀에 이상적

### WWA (Why-What-Acceptance)
**형식**: Why [전략적 맥락] → What [결과물] → Acceptance [기준]
**wwas** 스킬 적용:
- 전략적 이유 포함 (왜 이것을 만드는지)
- 독립적이고, 가치 있고, 테스트 가능한 아이템 작성
- 비즈니스 맥락이 필요한 크로스펑셔널 팀에 적합

## 워크플로우

### 1단계: 기능 수락

PRD, 기능 설명, 사용자 리서치 발견, 또는 구두 아이디어 어떤 형태로든 수락합니다. PRD가 제공된 경우, 분해할 요구사항을 추출합니다.

### 2단계: 형식 결정

호출 시 명시되지 않은 경우 물어봅니다:
- "팀에서 어떤 형식을 사용하나요? **사용자 스토리**, **잡 스토리**, 또는 **WWA**?"
- 확실하지 않다면 사용자 스토리를 기본값으로 권장

### 3단계: 기능 분해

- 기능을 5-15개의 독립적인 스토리로 분해 (하나의 스프린트 내 완료할 수 있을 만큼 작게)
- 각 스토리가 독립적으로 가치 있는지 확인 (그 자체로 사용자 가치 제공)
- 의존성 및 우선순위 순으로 정렬
- 스토리당 3-5개의 인수 기준 작성
- 디자인 입력이나 기술 스파이크가 필요한 스토리 표시

### 4단계: 스토리 생성

```
## 백로그: [기능 이름]

**형식**: [사용자 스토리 / 잡 스토리 / WWA]
**총 스토리 수**: [개수]
**예상 총 작업량**: [가능한 경우]

### 스토리

#### 스토리 1: [짧은 제목]
**[선택한 형식의 스토리]**

인수 기준:
- [ ] [기준 1]
- [ ] [기준 2]
- [ ] [기준 3]

우선순위: [P0/P1/P2] | 작업량: [S/M/L] | 의존성: [없음 또는 목록]

---
[각 스토리마다 반복]

### 스토리 맵
[시각적 순서: 반드시 있어야 함 → 있어야 함 → 있으면 좋음]

### 기술 노트
[공통 관심사: API 변경, 데이터 마이그레이션, 인프라]

### 미결 질문
[구현 시작 전에 답이 필요한 것들]
```

마크다운으로 저장합니다.

### 5단계: 다음 단계 제안

- "이 스토리들에 대한 **테스트 시나리오를 생성**해 드릴까요?"
- "개발 및 테스트를 위한 **더미 데이터를 작성**해 드릴까요?"
- "이 스토리들에 대한 **스프린트 용량을 추정**해 드릴까요?"
- "**다른 형식으로 변환**해 드릴까요? (사용자 스토리 ↔ 잡 스토리 ↔ WWA)"

## 참고사항

- 스토리 1개 = 배포 가능한 가치 단위 하나 — 다른 스토리가 있어야만 유용하다면 합쳐야 합니다
- 인수 기준은 추가적인 해석 없이 QA가 테스트 가능해야 합니다
- 에러 처리와 엣지 케이스는 해피 패스 스토리 내의 불릿 포인트가 아닌 별도의 스토리로 작성하세요
- 기능이 크다면 (스토리 15개 이상), 에픽이나 단계로 그룹화할 것을 제안하세요
- 스파이크(추정 전에 기술 조사가 필요한 경우)가 필요한 스토리를 표시하세요
