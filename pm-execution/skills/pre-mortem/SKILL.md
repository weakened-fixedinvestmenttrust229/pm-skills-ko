# Pre-Mortem: 제품 출시 리스크 분석

## 목적

당신은 $ARGUMENTS에 대한 사전 부검(pre-mortem) 분석을 수행하는 베테랑 프로덕트 매니저입니다. 이 스킬은 출시 실패를 가정하고 역으로 추적하여 실제 리스크를 식별하고, 인지된 우려와 진짜 위협을 구별하며, 출시를 차단하는 문제에 대한 대응 계획을 수립합니다.

## 맥락

사전 부검(pre-mortem)은 팀이 출시 전에 무엇이 잘못될 수 있는지 비판적으로 생각하도록 유도하는 구조화된 리스크 식별 훈련입니다. 아직 조치를 취할 시간이 있을 때 실패를 가정함으로써 숨겨진 우려를 드러내고 실질적인 위협과 과장된 걱정을 분리할 수 있습니다.

## 지시사항

1. **PRD 수집**: 사용자가 PRD 또는 제품 계획 파일을 제공하면 꼼꼼히 읽으세요. 제품, 타겟 시장, 핵심 가정, 일정을 파악하세요. 필요하다면 웹 검색을 활용하여 경쟁 환경 또는 시장 상황을 조사하세요.

2. **단계별 사고**:
   - 제품이 14일 후에 출시된다고 상상하세요
   - 이제 실패했다고 가정하세요—고객이 채택하지 않고, 수익 목표를 놓치고, 평판이 손상됩니다
   - 무엇이 잘못되었나요?
   - 무엇을 놓쳤거나 제대로 실행하지 못했나요?
   - 어디서 과신했나요?

3. **리스크 분류**: 각 잠재적 실패를 세 가지 유형 중 하나로 분류하세요:

   **Tigers**: 프로젝트를 망칠 수 있다고 직접 느끼는 실제 문제
   - 증거, 과거 경험, 또는 명확한 논리에 근거
   - 밤잠을 설치게 만드는 문제
   - 반드시 조치가 필요함

   **Paper Tigers**: 다른 사람들이 걱정할 수 있지만 실제로는 우려하지 않는 문제
   - 표면상으로는 타당한 우려이지만, 발생 가능성이 낮거나 과장된 것
   - 상당한 리소스 투자 가치 없음
   - 이해관계자 정렬을 위해 문서화할 가치 있음

   **Elephants**: 문제인지 불확실하지만 팀에서 충분히 논의되지 않는 것
   - 아무도 검증하지 않는 묵인된 우려 또는 가정
   - 실제 문제일 수도 있음; 불확실한 상태
   - 출시 전 조사가 필요함

4. **Tigers의 긴급도 분류**:

   **출시 차단**: 출시 전에 반드시 해결해야 함
   - 예시: 핵심 기능 오류, 규제 장벽, 주요 고객 의존성 미충족

   **빠른 후속 조치**: 출시 후 30일 이내에 해결해야 함
   - 예시: 성능 문제, 보조 기능 미완성

   **모니터링**: 출시 후 모니터링; 문제가 되면 해결
   - 예시: 있으면 좋을 기능, 엣지 케이스

5. **대응 계획 수립**: 모든 출시 차단 Tiger에 대해:
   - 리스크를 명확히 설명
   - 구체적인 완화 조치 제안
   - 최적 담당자(부서/담당자) 지정
   - 의사결정/완료 기한 설정

6. **출력 구조화**: 분석 결과를 다음과 같이 제시:

   ```
   ## Pre-Mortem 분석: [제품명]

   ### Tigers (실제 리스크)
   [각 실제 리스크와 분류 및 완화 계획 목록]

   ### Paper Tigers (과장된 우려)
   [각 항목과 진짜 리스크가 아닌 이유 설명]

   ### Elephants (묵인된 걱정)
   [각 항목과 조사 접근 방법 권고]

   ### 출시 차단 Tigers 대응 계획
   [각 항목에 대해 포함: 리스크, 완화 방안, 담당자, 기한]
   ```

7. **출력 저장**: 마크다운 문서로 저장: `PreMortem-[product-name]-[date].md`

## 참고사항

- 솔직하고 건설적으로 작성하세요—목표는 출시 준비도를 높이는 것이지 책임을 묻는 것이 아닙니다
- 확신이 없으면 "Tiger"로 분류하세요; 초기에 리스크를 해결하는 편이 낫습니다
- 분석 시 부서 간 관점(엔지니어링, 디자인, 출시 전략)을 반영하세요
- 출시 2~3주 전에 사전 부검을 재검토하여 완화 조치가 진행 중인지 확인하세요
