# /setup-metrics -- 프로덕트 지표 대시보드 설계

제품 또는 기능을 위한 포괄적인 지표 프레임워크를 설계합니다 — 올바른 북극성 지표 선정부터 문제를 조기에 감지하는 알림 임계값 정의까지.

## 실행 방법

```
/setup-metrics SaaS project management tool
/setup-metrics New checkout flow we just launched
/setup-metrics             # 무엇을 측정할지 질문합니다
```

## 워크플로우

### 1단계: 측정 대상 파악

사용자에게 질문합니다:
- 어떤 제품 또는 기능 영역의 지표를 설정하나요?
- 어떤 단계에 있나요? (출시 전, 최근 출시, 성숙 단계)
- 현재 비즈니스 목표나 OKR은 무엇인가요?
- 기존 지표가 있나요? 무엇이 빠지거나 문제가 있나요?
- 어떤 분석 도구를 사용하고 있나요? (구현 조언을 맞춤화하는 데 도움이 됩니다)

### 2단계: 지표 프레임워크 정의

**metrics-dashboard** 스킬을 적용합니다:

**북극성 지표(North Star Metric):**
- 제품이 사용자에게 전달하는 가치를 가장 잘 포착하는 단일 지표 식별
- 기준에 따라 검증: 가치 전달을 측정하는지, 선행 지표인지, 실행 가능한지
- 지표를 정확하게 정의 (공식, 데이터 소스, 시간 창)

**입력 지표(Input Metrics) (3-5개):**
- 북극성 지표를 구동하는 레버 식별
- 각 입력 지표는 팀이 직접 실행할 수 있어야 합니다
- 인과 관계 연결: 입력 → 북극성 → 비즈니스 결과

**건강 지표(Health Metrics) (3-5개):**
- 안정적으로 유지되어야 하는 지표 — 저하되면 문제가 있는 것입니다
- 예: 오류율, 지연 시간, 지원 티켓 건수, NPS, 이탈률
- "정상" 범위와 저하 임계값 정의

**역지표(Counter-Metrics) (1-2개):**
- 잘못된 방향으로 최적화하고 있는지를 나타낼 수 있는 지표
- 예: 북극성이 "일일 활성 사용자(DAU)"라면, 역지표는 빈 참여를 방지하기 위한 "세션 품질"

### 3단계: 알림 임계값 설계

각 지표에 대해:

| 지표 | 초록 | 노랑 | 빨강 | 확인 주기 |
|--------|-------|--------|-----|----------------|
| [지표] | [정상 범위] | [경고] | [위험] | [일/주] |

- **노랑**: 조사 필요 — 뭔가 잘못될 수 있음
- **빨강**: 즉시 조치 — 담당자에게 알리거나 에스컬레이션

### 4단계: 대시보드 스펙 작성

```
## 지표 대시보드: [제품/기능]

**북극성 지표**: [지표 이름]
**정의**: [정확한 공식]
**현재 값**: [알고 있는 경우]
**목표**: [목표치]

### 입력 지표
| 지표 | 정의 | 담당자 | 목표 | 현재 |
|--------|-----------|-------|--------|---------|

### 건강 지표
| 지표 | 정상 범위 | 노랑 임계값 | 빨강 임계값 |
|--------|-------------|-----------------|---------------|

### 역지표
| 지표 | 중요한 이유 | 주의할 것 |
|--------|---------------|-----------|

### 지표 트리
북극성: [지표]
├── 입력: [지표 1] → [팀/행동]에 의해 구동
├── 입력: [지표 2] → [팀/행동]에 의해 구동
├── 입력: [지표 3] → [팀/행동]에 의해 구동
└── 역지표: [지표] → [저하 신호] 주의

### 구현 메모
- 데이터 소스: [각 지표가 어디서 오는지]
- 갱신 주기: [실시간 / 시간별 / 일별]
- 도구 추천: [사용자의 스택에 맞게]

### 검토 주기
- **일별**: 북극성 지표 및 건강 지표 확인
- **주별**: 입력 지표 트렌드 검토, 팀 스탠드업에서 논의
- **월별**: 심층 분석 — 입력이 예상대로 북극성을 구동하고 있는지?
- **분기별**: 지표 프레임워크 자체 재평가
```

사용자의 작업 공간에 마크다운 파일로 저장합니다.

### 5단계: 다음 단계 제안

- "이 지표를 계산하는 **SQL 쿼리를 작성**할까요?"
- "이 지표 프레임워크를 기반으로 **OKR을 작성**할까요?"
- "현실적인 기준선을 설정하기 위한 **코호트 분석을 구축**할까요?"
- "**주간 지표 검토 템플릿을 설정**할까요?"

## 참고 사항

- 좋은 북극성 지표는 드뭅니다 — 대부분의 팀은 허영 지표를 선택합니다. 단순한 참여가 아닌 *사용자에게 전달된 가치*를 포착하는 지표를 선택하세요
- 입력 지표는 북극성을 설명하는 데 있어 MECE(상호 배타적, 전체 포괄적)해야 합니다
- 제품이 출시 전이라면 지금 지표를 정의하되, 출시 후 기준선 보정이 필요할 것임을 명시합니다
- 역지표는 굿하트의 법칙을 방지합니다 — 지표가 목표가 되면, 좋은 지표이기를 그칩니다
- 아무도 확인하지 않는 복잡한 대시보드보다 잘 계측된 소수의 지표로 시작하는 것을 권장합니다
