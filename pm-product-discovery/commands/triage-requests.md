# /triage-requests -- 기능 요청 트리아지

지원 티켓, 영업 통화, 설문조사, 또는 Slack에서 쌓인 기능 요청을 우선순위가 정해진 실행 가능한 백로그로 전환합니다.

## 실행 방법

```
/triage-requests                           # 입력을 요청합니다
/triage-requests [paste a list of requests]
/triage-requests [upload a CSV/spreadsheet]
```

## 워크플로우

### 1단계: 기능 요청 수령

어떤 형식으로든 요청을 수령합니다:
- **붙여넣기 텍스트**: 한 줄 또는 단락 단위로 나열된 요청 목록
- **업로드 파일**: 요청 데이터가 담긴 CSV, Excel, 또는 텍스트 파일
- **구조화된 데이터**: 입력에 열(요청자, 요청, 날짜 등)이 있으면 유지

입력이 없으면 사용자에게 기능 요청을 붙여넣거나 업로드해달라고 요청합니다.

각 요청을 파싱하여 다음을 추출합니다:
- 핵심 요구 사항 (사용자가 원하는 것)
- 맥락 (누가 언제 왜 요청했는지 — 가능한 경우)
- 빈도 신호 (비슷한 것을 요청한 사람 수)

### 2단계: 우선순위 결정 맥락 수집

사용자에게 질문합니다 (한꺼번에 묻지 않고 대화하듯):
- 제품은 무엇인가요? 어떤 단계에 있나요?
- 현재 전략적 목표나 OKR은 무엇인가요? (정렬 평가에 도움)
- 고려해야 할 제약이 있나요? (팀 규모, 기술 부채, 임박한 마감)
- 더 가중치를 두어야 할 세그먼트가 있나요? (엔터프라이즈, 이탈 사용자, 파워 유저)

### 3단계: 분류 및 분석

**analyze-feature-requests** 스킬을 적용합니다:

- **테마 클러스터링**: 유사한 요청을 테마로 그룹화 (예: "리포팅 및 분석", "협업", "모바일 경험")
- **테마별 요청 수**: 각 테마에 매핑되는 고유 요청 수
- **전략적 정렬**: 각 테마를 명시된 목표 대비 평가 (높음/중간/낮음/없음)
- **세그먼트 분석**: 어떤 사용자 세그먼트가 어떤 테마를 이끌고 있는지
- **감정 신호**: 요청에 좌절, 이탈 위협, 또는 기쁨이 동반되는지

### 4단계: 우선순위 결정

**prioritize-features** 스킬을 적용합니다:

각 테마(및 각 테마 내 상위 개별 요청)에 대해:

| 요소 | 평가 |
|--------|-----------|
| **영향** | 몇 명의 사용자가 영향을 받나요? 얼마나 심각한가요? |
| **전략적 정렬** | 현재 목표에 부합하나요? |
| **노력 추정** | 티셔츠 사이즈 (S/M/L/XL) |
| **위험** | 이것을 하지 않으면 어떻게 되나요? |
| **매출 신호** | 거래, 유지, 또는 확장과 연결되어 있나요? |

테마 순위를 매기고 우선순위화된 목록을 작성합니다.

### 5단계: 트리아지 보고서 생성

```
## 기능 요청 트리아지 보고서

**날짜**: [오늘]
**분석된 요청**: [건수]
**식별된 테마**: [건수]

### 테마 요약
| # | 테마 | 요청 수 | 주요 요구 | 정렬 | 영향 | 노력 | 우선순위 |
|---|-------|----------|---------|-----------|--------|--------|----------|

### 우선순위 1: 즉시 실행
[단기 계획에 포함할 테마/요청]
- **[테마]**: [X]건 요청 — [긴급한 이유]
  - 주요 요청: [목록]
  - 권장 조치: [빌드 / 프로토타입 / 조사]

### 우선순위 2: 다음 계획
[긴급하지는 않지만 계획할 가치가 있는 테마]

### 우선순위 3: 추가 신호 수집
[잠재력은 있지만 증거가 불충분한 테마]

### 우선순위 4: 거절 또는 연기
[전략과 맞지 않는 요청 — 근거와 함께]

### 주목할 만한 개별 요청
[테마로 묶이지 않은 고가치 단독 요청]

### 패턴 및 인사이트
- [사용자가 전달하는 핵심 인사이트]
- [세그먼트별 패턴]
- [사용자가 요청하는 것과 근본적인 니즈 사이의 차이]
```

보고서를 사용자의 작업 공간에 마크다운 파일로 저장합니다.

### 6단계: 다음 단계 제안

- "최우선 항목에 대한 **사용자 스토리를 작성**할까요?"
- "이 테마들에 대한 **솔루션을 브레인스토밍**할까요?"
- "빌드 전에 수요를 검증하기 위한 **실험을 설계**할까요?"
- "이 분석을 요약한 **이해관계자 업데이트 초안을 작성**할까요?"

## 참고 사항

- 사용자가 열이 있는 CSV를 제공하면 데이터 구조를 유지하고 보강합니다
- 요청 뒤에 있는 니즈를 찾습니다 — "다크 모드 추가"는 사실 "긴 세션 중 눈의 피로 감소"를 의미할 수 있습니다
- 서로 충돌하는 요청을 표시합니다 (예: "UI 단순화" vs. "설정 옵션 추가")
- 요청 건수가 많은 경우(50+), 먼저 테마를 요약하고 특정 테마에 대한 심층 분석을 제안합니다
- 입력이 구조화된 데이터인 경우 보강된 데이터를 다운로드 가능한 CSV로 출력합니다
